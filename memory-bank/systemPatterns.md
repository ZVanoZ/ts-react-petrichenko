# System Patterns

### Общие принципы

- **Строгая типизация**:
  - избегать `any`, по возможности использовать точные или обобщённые типы;
  - описывать контракты через `interface`/`type` с понятными именами.

- **ООП и композиция**:
  - в разделах про классы — демонстрировать наследование, интерфейсы и модификаторы доступа;
  - в React-практике — предпочитать композицию компонентов и hooks вместо избыточной вложенности логики.

- **Чистые модули / маленькие файлы**:
  - уроки и примеры держать изолированными по темам (как уже сделано: отдельные директории под конкретные уроки).

### Паттерны для AI-взаимодействия

- Ссылаться на файлы/директории явно (`20-ts/30-generics-manipulation/050-generics-classes`), чтобы уменьшать объём контекста.
- При доработке кода:
  - сначала уточнять цель (что хотим показать/отработать на примере);
  - затем просить о точечных изменениях, не перегенерируя весь файл без необходимости.

### Workflow: проверка актуальности уроков (@TODO "Cursor AI")

Основан на блоке `@TODO: "Cursor AI"` в корневом `readme.md`.

1. **Проверка одного урока**
   - Вход: путь к уроку (например, `20-ts/30-generics-manipulation/050-generics-classes`), краткое описание темы.
   - Действия:
     - Использовать MCP `user-context7` (связка `resolve-library-id` → `query-docs`) для соответствующей технологии (TypeScript или React).
     - Сравнить темы урока с актуальной документацией (основные изменения в языке/фреймворке, добавленные/обесцененные практики).
   - Выход:
     - **Краткий результат** — добавить/обновить сжатое резюме по актуальности в корневом `readme.md` (отдельный подпункт или список).
     - **Подробный результат** — создать/обновить файл `ai-relevance.md` в директории урока:
       - отметить, что осталось актуальным;
       - что устарело или стало спорным;
       - какие дополнительные темы стоит изучить (по ссылкам из актуальной документации).

2. **Проверка всех существующих уроков по запросу пользователя**
   - По запросу пользователя обойти все директории уроков внутри выбранного раздела (или всего репозитория).
   - Для каждой директории урока выполнить шаги из пункта 1.
   - В корневом `readme.md` поддерживать сводный список уроков с пометками по актуальности.

3. **Анализ выполненного задания**
   - По запросу пользователя:
     - прочитать файлы решения в директории урока;
     - сопоставить с типичной/эталонной реализацией (при необходимости используя внешний репозиторий ответов курса);
     - в `ai-relevance.md` или отдельном `ai-review.md` указать:
       - пробелы в знаниях;
       - неточные/опасные решения;
       - рекомендации по доработке.

Этот workflow описывается здесь (в `systemPatterns.md`) и может быть дополнительно отражён в правилах `.cursor`, но **истиной по умолчанию** считается именно это описание.
